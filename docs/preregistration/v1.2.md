## §1. Research Question

Do task-specific parameter updates in pretrained LLMs lie in a shared low-dimensional subspace, indicating that adaptation is primarily a control problem over a pretrained capability substrate rather than knowledge injection?

---

## §2. Central Hypothesis

Task adaptation in pretrained LLMs is primarily a **control problem** over a pretrained capability substrate, not a knowledge injection problem.

---

## §3. Sub-Hypotheses

### H1a — Shared Adaptive Subspace (Primary)

Task-specific update vectors concentrate in a shared low-dimensional subspace.

**Definitions:**

Let:
```
U = [u_1, u_2, ..., u_N] ∈ ℝ^(D × N)
```
where `u_i` is the normalized update vector for task `i`.

Compute SVD:
```
U = V Σ W^T
```

Define 90%-energy rank:
```
r_90 = min{ k : Σ_{i=1}^{k} σ_i² / Σ_{i=1}^{N} σ_i² ≥ 0.9 }
```

Define effective dimension:
```
D_eff = dimension of u_i (total number of parameters in update vector)
```

Primary ratio:
```
ρ = r_90 / min(N, √D_eff)
```

**Decision rule with confidence intervals:**

| Tier | ρ range | Bootstrap 95% CI requirement |
|------|---------|-------------------------------|
| Strong pass | ρ < 0.30 | CI upper bound < 0.35 |
| Marginal pass | 0.30 ≤ ρ < 0.50 | CI overlaps 0.30-0.50 |
| Inconclusive | 0.50 ≤ ρ < 0.70 | — |
| Fail | ρ ≥ 0.70 | CI lower bound ≥ 0.65 |

Bootstrap procedure: 1000 samples, resample tasks with replacement.

**Note:** If N < 10, the experiment does not proceed to H1a evaluation. Expand task coverage first.

---

### H1b — Directional Separability Within Shared Subspace (Primary)

Task directions remain separable **after projection** onto the shared subspace.

**Procedure (to avoid circularity with H1a):**

1. From H1a SVD, extract top-k basis: `V_k` where `k = r_90`
2. Project each update vector: `ũ_i = V_k V_k^T u_i`
3. Compute all pairwise cosine similarities: `cos(ũ_i, ũ_j)` for all `i ≠ j`

**Decision rule:**

| | Pass | Fail |
|-|------|------|
| Condition | mean |cos| < 0.30 AND max |cos| < 0.60 | mean |cos| > 0.50 |
| Gray zone | 0.30 ≤ mean |cos| ≤ 0.50 | inconclusive |

---

### H1c — Semantic Alignment (Exploratory)

Projected task directions align with interpretable features (e.g., SAE-derived features or known circuits from mechanistic interpretability research).

**Procedure:**
- For each `ũ_i`, identify top-k SAE features with maximum activation overlap
- Compute IoU between observed features and interpretable ground truth (if available)

**Random baseline:**
- Generate 1000 random unit vectors in `V_k` subspace
- Compute same IoU statistic for each
- Random baseline = 95th percentile of this distribution

**Decision rule:**
- Pass: Observed IoU > random baseline 95th percentile (p < 0.05)
- Otherwise: No semantic claim is allowed

**Note:** H1c is exploratory. Results are reported separately and do not affect the central claim determination.

---

### H2 — Transferability (Primary)

Task-specific updates transfer across similar base models.

**Metric:**
```
Retention = Performance_transfer / Performance_original
```

**Model pairs (pre-specified):**

| Setting | Source → Target | Type |
|---------|----------------|------|
| Primary | Qwen2.5-1.5B-Instruct → Qwen2.5-1.5B (base) | Same family, different post-training |
| Secondary A | Qwen2.5-1.5B → Qwen2.5-3B (if compute allows) | Same family, different size |
| Secondary B | Qwen2.5-1.5B → TinyLlama-1.1B | Cross-family |

**Decision rule (primary setting only):**

| | Pass | Fail |
|-|------|------|
| Condition | Retention > 70% | Retention < 30% |
| Gray zone | 30% ≤ Retention ≤ 70% | inconclusive |

Secondary settings are reported as exploratory.

---

### H3 — Statistical Self-Similarity (Exploratory)

Normalized layer weight distributions show statistically lower Wasserstein distance than random/shuffled baselines.

- Random baseline: shuffle layer assignments and recompute Wasserstein distances
- Pass: observed distribution of W-distances is statistically lower than shuffled baseline (Mann-Whitney U, p < 0.05)

H3 is exploratory. Failure does not falsify the central claim.

## §4. Methodology

### §4.1 Base Models

**Primary:**
- Qwen2.5-1.5B-Instruct (open-source, fits RTX 4090 for GRPO)

**Secondary (for H2 transfer):**
- Qwen2.5-1.5B (base, non-instruct)
- TinyLlama-1.1B-Chat (cross-family)
- Qwen2.5-3B-Instruct (size-scaled, if cloud compute available)

All model choices are fixed before data collection begins.

---

### §4.2 Task Distribution

**Primary task distribution:**
```
T_reasoning = {
  GSM8K,         # Elementary math
  MATH,          # Competition math
  AIME,          # Hard competition math
  AMC,           # Multiple choice math
  MATH500,       # MATH subset
  HumanEval,     # Code generation
  MBPP,          # Code completion
  CommonsenseQA, # Common sense reasoning
  HellaSwag,     # Sentence completion
  ARC-Challenge  # Science QA
}
```

- **Minimum N:** 10 tasks (all above)
- **Preferred extension:** 15–20 tasks (additional coding/language tasks)
- **Exclusion criterion:** Tasks where GRPO training is infeasible (see §4.5)

---

### §4.3 Update Vector Definition

**Primary object (registered):**
```
u_i = concat(ΔW_A, ΔW_B) over all LoRA adapter layers
```
where ΔW_A, ΔW_B are the low-rank adapter matrices after training.

**Secondary object (exploratory):**
```
u_i = vec(ΔW) restricted to:
  - Attention: Q, K, V, O matrices
  - FFN: gate, up, down matrices
  (not all layers — representative subset only)
```

Secondary analysis is reported separately, labeled as exploratory.

---

### §4.4 Normalization

Two separate analyses are pre-registered, conducted on distinct data objects:

**§4.4a — Direction Analysis** (used for H1a, H1b)
```
u_i ← u_i / ||u_i||
```
All direction-based analysis uses these normalized vectors.

**§4.4b — Magnitude Analysis** (used for B_adaptive estimation)
```
u_i is kept unnormalized
||u_i|| is recorded for each task
```
Magnitude-based analysis reports: ||u_i||, parameter count, effective update size per task. This measures B_adaptive[T] directly.

---

### §4.5 Training Protocol

**Primary method:** GRPO (Group Relative Policy Optimization)

**Fixed hyperparameters:**
- Gradient steps per task: 1000 (fixed, no early stopping)
- Optimizer: AdamW
- Learning rate: 1 × 10⁻⁴
- LoRA rank: 8
- LoRA alpha: 16
- Target modules: q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj

**Baseline (secondary, exploratory):**
- SFT (supervised fine-tuning) with same hyperparameters
- Reported separately, not used for primary registered analysis

**Task exclusion criteria (pre-specified):**

A task is excluded from primary registered analysis if ANY of:
1. GRPO training loss diverges within first 100 steps (> 2× initial loss and not recovering)
2. Reward signal is absent: reward = 0 for > 80% of rollouts at step 200
3. No improvement over base model after 1000 steps (0% delta on task metric)

All exclusions must be logged with:
- Task name
- Diagnostic plots (loss curve, reward curve)
- Exclusion criterion triggered

Excluded tasks are NOT substituted with SFT equivalents in the primary analysis.

## §5. Analysis Plan

### §5.1 Registered Primary Analyses

In order of execution:

1. **H1a:** Compute ρ from SVD of normalized update matrix; determine tier (strong/marginal/inconclusive/fail)
2. **H1b:** Only if H1a ≠ fail; project updates, compute pairwise cosine similarities
3. **H2:** Conduct transfer experiments on primary model pair; compute Retention

These three analyses determine the central claim tier (see §7 Interpretation Rules).

### §5.2 Exploratory Analyses

Conducted after primary analyses; reported in separate section of paper:

- H1c: Semantic alignment (requires SAE features, if available)
- H3: Layer self-similarity (Wasserstein distances)
- Intrinsic dimension estimation (TwoNN algorithm, multi-scale)
- Magnitude trends: ||u_i|| as function of task difficulty

Exploratory analyses must be explicitly labeled as exploratory in all presentations and publications.

## §6. Stopping Rules

**Minimum data requirement:**
- Registered analysis begins only after N ≥ 10 tasks have complete update vectors

**Bootstrap stopping criterion:**
- After initial N tasks, compute bootstrap 95% CI for ρ
- Stop collecting additional tasks when CI width < 0.05
- If CI width remains > 0.05 after N = 20 tasks, report inconclusive for H1a

**Computational stopping criterion:**
- Maximum compute budget: to be logged explicitly before experiment execution
  (Best effort: 200 GPU-hours on RTX 4090)
- If budget is reached before N = 10, report Phase 1 incomplete and redesign

## §7. Interpretation Rules

Central claim tier is determined by the joint outcome of H1a, H1b, H2:

| H1a | H1b | H2 | Allowed Central Claim |
|-----|-----|----|-----------------------|
| Fail | N/A | Any | No shared adaptive structure detected |
| Inconclusive | N/A | Any | Weak evidence; inconclusive |
| Marginal pass | Fail | Any | Possible shared update manifold only |
| Marginal pass | Pass | Fail | Suggestive capability-like structure, no transfer |
| Marginal pass | Pass | Pass | Suggestive capability-like decomposition with transfer |
| Strong pass | Fail | Any | Shared subspace exists, non-separable |
| Strong pass | Pass | Fail | Capability-like decomposition, limited transfer |
| Strong pass | Pass | Pass | **Capability-like decomposition with transfer** (target) |
| All above + H1c | Pass | Pass | Interpretable capability substrate |

**Rule:** The paper's abstract and conclusion must use only the language corresponding to the achieved tier.

---

## §8. Falsification Rules

### Hard Falsification

The framework's central claim is **strongly disfavored** if ALL of:
- H1a: Fail (ρ ≥ 0.70 with CI lower bound ≥ 0.65)
- H2: Fail (Retention < 30%)

In this case, proceed to Recovery Path (§9).

### Soft Falsification

| Outcome | Required Adjustment |
|---------|---------------------|
| H1a pass, H1b fail | Claim "shared update manifold" only; remove "capability-like" language |
| H1b pass, H1c fail | No interpretability claim permitted |
| H3 fail | No effect on central claim (H3 is exploratory) |

---

## §9. Recovery Path

If hard falsification is triggered, the alternative hypothesis for the paper becomes:

> **Tiny updates may work through decision-boundary smoothing, implicit regularization, or output distribution reshaping, rather than structured adaptive subspaces.**

Alternative paper title:
*"Understanding TinyLoRA Without Shared Adaptive Subspaces"*

This is still a publishable, valuable empirical contribution.

The alternative paper would investigate:
- What changes in the loss landscape after tiny RL updates?
- Does output probability distribution shift in a structured way?
- Is there implicit L2 regularization that explains the small update size?
